01

Point Platform · Event Reliability

발행 실패와 처리 실패는 복구 방법부터 달랐습니다.

포인트 차감, 회수, 만료 알림이 늦게 실패하면 정산 확인과 고객 응대가 함께 길어집니다. 어디서 멈췄는지 기록하고, 다음 행동이 다른 실패는 별도 상태로 남겼습니다.

담당
이벤트 상태·수동 복구·정체 알림 설계 및 구현
구조
내부 이벤트 9종, 상태 6개, 타입별 복구 전략 8개
저장
PostgreSQL internal_event_record · JSONB payload
관측
IN_PROGRESS 15분 초과 건을 타입별로 집계해 Slack 전송
상황

성공 여부만으로는 다음 행동을 정할 수 없었습니다.

메시지를 보내지 못한 경우와 소비자가 처리하다 실패한 경우는 확인할 데이터와 재시도 위치가 다릅니다. 한 상태로 뭉치면 운영자가 로그와 원장을 다시 대조해야 했습니다.

판단

실패 위치를 상태로 남기고 복구 작업은 타입별로 분리했습니다.

생성부터 복구 불가 실패까지 상태를 나누고, 포인트 차감·회수·만료 알림처럼 조건이 다른 작업은 각각의 수동 복구 전략으로 다시 실행했습니다.

이벤트 처리와 복구 구조 전체 보기
운영 흐름

발행 실패, 처리 실패, 복구 불가 실패를 상태로 남기고 수동 복구와 15분 정체 알림을 별도 경로로 운영했습니다.

세부 구현과 예외 처리
  • 내부 이벤트 9종을 생성, 발행 실패, 처리 중, 완료, 처리 실패, 복구 불가 실패의 6개 상태로 기록했습니다.
  • 수동 복구 API가 타입과 상태로 레코드를 찾고 저장된 payload로 8개 전략 중 하나를 실행하도록 구성했습니다.
  • Kafka에 다시 넣지 않고 타입별 업무 서비스를 직접 호출해 복구 위치를 고정했습니다.
  • IN_PROGRESS 상태로 15분 이상 멈춘 추적 대상 8종만 조회해 타입별 건수를 Slack에 전송했습니다.
  • 상태 전환 타이밍 오류, 잘못된 상태 검사, 이벤트 발송 실패 로깅 누락을 수정했습니다.
  • 배치 재시도와 프록시 기반 재시도 호출 문제를 확인하고 포인트 선점 복구 경로를 보강했습니다.
02

Point Ledger · Balance / Archive

대량 처리 단위와 정산 검증 순서를 함께 바꿨습니다.

원장 소멸, 만료 안내, 아카이브는 서로 다른 배치입니다. 소멸 배치는 원장·잔액·거래 내역을 갱신하고, 안내 배치는 알림 레코드와 발송 이벤트를, 아카이브 배치는 보관 테이블의 이동 순서를 각각 관리합니다.

담당
원장 소멸 paging 전환·아카이브 배치 구현·만료 알림 이벤트 개선
처리 단위
전체 적재 대신 paging 기반 반복 처리
보관 대상
point_core 6개·point_service 4개 archive 테이블
동시성
분산락, 예산 경합, 유료·무료 원장 혼합 사용
상황

세 Job은 같은 원장과 거래 데이터를 서로 다른 시점에 다룹니다.

전체 데이터를 메모리에 올리는 방식과 느슨한 처리 경계는 데이터가 많아질수록 실행 부담과 확인 범위를 키웁니다.

판단

처리량보다 같은 단위로 다시 돌릴 수 있는 구조를 먼저 잡았습니다.

원장 조회와 소멸은 페이지 단위로 반복하고, 만료 대상 수집부터 이벤트 발행과 아카이브까지의 순서를 테스트에 남겼습니다.

원장 배치 구조 전체 보기
처리 순서

소멸, 만료 안내, 아카이브를 독립된 배치로 나누고 각 Job의 조회 단위와 후속 저장 순서를 테스트에 남겼습니다.

정합성과 회원 식별 처리
  • 만료 대상 수집, 날짜 계산, 기업 회원 필터, 만료 알림 레코드 batch insert 순서를 검증했습니다.
  • point_core 6개와 point_service 4개 테이블의 아카이브 흐름과 테스트를 추가했습니다.
  • 동일·상이한 lock key의 동시 요청과 같은 예산 ID의 동시 차감 시나리오를 검증했습니다.
  • 유료·무료 원장 혼합 사용 케이스를 별도로 검증했습니다.
  • 회원-워크스페이스 매핑, master/member 분리, 회원 탈퇴 이벤트, globalUserRef 중복 처리를 보강했습니다.
03

Feed & Search · Serving / Retrieval

빠른 캐시보다 언제 다시 계산할지가 더 중요했습니다.

같은 사용자가 짧은 시간에 여러 페이지를 읽는 피드는 매번 검색하면 비싸고, 캐시를 오래 믿으면 낡습니다. page queue, 후보 풀, 요청 지문, 노출 이력을 따로 둬 재사용 조건을 명시했습니다.

후보 조회
OpenSearch 9개 소스 병렬 실행, 설정상 최대 2,700건
랭킹
FPR 500 → SPR/Re-Rank 100 → 페이지 20건
캐시
Snapshot · FPR · SPR · User Vector, TTL 30분
분기
요청 지문 변경, guest → member, Redis 장애
상황

호출 비용과 추천 신선함이 같은 캐시 정책 안에서 충돌했습니다.

좌표나 필터가 바뀌었는데 이전 후보를 계속 쓰면 빠르지만 틀린 응답이 됩니다. 반대로 페이지마다 전부 다시 계산하면 검색과 랭킹 계층을 반복해서 탑니다.

판단

스크롤, 큐 소진, 조건 변경, 장애를 서로 다른 경로로 보냈습니다.

정상 스크롤은 page queue에서 끝내고, 후보가 부족할 때만 refill합니다. 좌표·필터 지문이 달라지면 후보 풀을 비우고, Redis 장애에는 응답을 우선했습니다.

피드 캐시와 장애 우회 구조 전체 보기
캐시 정책

cache hit, queue refill, 요청 조건 변경, 로그인 전환, Redis 장애에 각각 다른 재사용·무효화·우회 규칙을 적용했습니다.

랭킹·캐시·검색 정책 상세
  • Snapshot, FPR pool, SPR pool, user vector, global seen을 서로 다른 Redis key와 TTL로 관리했습니다.
  • 9개 OpenSearch 소스를 virtual thread에서 병렬 호출하고 한 소스의 실패나 timeout은 빈 후보로 접어 나머지 결과를 유지했습니다.
  • Snapshot에 남은 후보가 있으면 20건만 꺼내고, 소진됐을 때만 100건짜리 랭킹 window를 다시 채웠습니다.
  • seen은 노출한 페이지마다 기록하고 FPR에서 감점 신호로 사용했습니다.
  • 세션과 후보 캐시는 30분, 회원 global seen은 24시간 동안 유지했습니다.
  • 위치 기준 거리 가중치와 10km 반경, 좌표가 없는 공고의 보존·제외 정책을 반영했습니다.
  • 메인 4축 필터를 색인과 질의 조립에 반영하고 회원·비회원에서 같은 의미로 적용했습니다.
  • 유료 후보는 별도 영역으로 떼지 않고 FPR 점수 가산에 포함했습니다.
  • 재게시된 같은 공고는 FPR에서 contentHash 기반으로 다시 접었습니다.
  • 검색 인덱스 전환, read/write 계정 경계, strict mapping 이슈를 함께 다뤘습니다.
