01

실패 단계를 기록해 필요한 업무만 다시 실행했습니다

담당
이벤트 상태 기록·수동 복구·정체 알림 경로 설계·개발
구현
Kafka 이벤트의 발행·처리 상태를 단계별로 구분
저장
PostgreSQL에 처리 상태 기록 · JSONB에 복구용 payload 저장
알림
정체된 IN_PROGRESS 이벤트를 유형별로 묶어 Slack 알림
상황

같은 실패 상태로는 복구 지점을 정할 수 없었습니다

발행 전에 멈췄는지, 발행 후 업무 처리에서 멈췄는지에 따라 확인할 데이터와 재실행 범위가 달랐지만 하나의 실패 상태만 남았습니다.

바꾼 점

실패 단계와 업무를 분리해 복구 범위를 좁혔습니다

발행 실패와 처리 실패를 나눠 기록하고 payload를 보존했습니다. 포인트 차감·회수·만료 알림 가운데 완료되지 않은 업무만 골라 다시 실행했습니다.

02

실제 영속성 컨텍스트에서 JPA 변경 감지 오류를 재현하고 약 200만 건을 보정했습니다

Mock 테스트에서 놓친 저장 오류를 실제 영속성 컨텍스트에서 재현했습니다. 보정 스크립트로 잘못 저장된 값을 바로잡아 저장 데이터를 다시 신뢰할 수 있게 했습니다.

문제
JPA 변경 감지 오류
재현
실제 영속성 컨텍스트에서 저장 과정 검증
보정
오류 데이터 식별 · 보정 스크립트 적용
회귀 검증
실제 저장 동작을 확인하는 테스트 보강
상황

Mock 테스트만으로는 실제 저장 오류를 찾을 수 없었습니다

기존 테스트는 실제 영속성 컨텍스트의 변경 감지와 저장 시점을 거치지 않아 오류가 드러나지 않았습니다.

바꾼 점

실제 저장 환경에서 재현한 뒤 보정 경로를 만들었습니다

오류 원인을 재현해 확인하고 보정 스크립트를 작성했습니다. 잘못 저장된 값을 바로잡아 저장 데이터를 다시 신뢰할 수 있게 했습니다.

03

중단된 배치를 이어 처리하고 포인트 정합성을 검증했습니다

배치마다 재시작 기준을 분리하고, 동시 요청이 끝난 뒤 잔액·원장·예산을 함께 대조했습니다.

담당

보관 대상
참조 순서대로 옮긴 아카이브 테이블 10개
동시성
9개 API · 13개 시나리오 · 3·5·10개 동시 요청
상황

배치마다 중단 후 이어갈 지점이 달랐습니다

소멸 배치는 잔액·원장을 바꾸고 만료 알림 배치는 발송 대상을 모았습니다. 아카이브 배치는 참조 순서를 지켜 테이블을 옮겨야 했습니다.

바꾼 점

진행 위치를 따로 기록하고 결과 정합성을 확인했습니다

배치별 진행 위치를 분리해 재실행 범위를 제한했습니다. 동시 요청 후에는 잔액·원장·예산·후속 이벤트를 함께 대조했습니다.
