Point Ledger · Batch / Archive

원장 소멸은 페이지 단위로 처리하고, 10개 아카이브 테이블은 이관 순서까지 테스트했습니다.

소멸, 만료 안내, 아카이브는 같은 원장 데이터를 다루지만 시작 시점과 후속 작업이 다릅니다. 한 번에 읽고 쓰던 범위를 줄이고 각 작업이 어디까지 진행됐는지 확인할 수 있게 나눴습니다.

  • 포인트 소멸 작업을 한 번에 처리하는 Tasklet에서 페이지 기반 Chunk 처리로 바꿔 조회와 쓰기 단위를 분리했습니다.
  • 만료 대상 수집, 날짜 계산, 기업 회원 필터, 알림 레코드 일괄 저장, Kafka 발행 순서를 테스트로 고정했습니다.
  • point_core 6개와 point_service 4개 등 10개 아카이브 테이블을 정해진 순서로 이관하는 operator 구성을 구현했습니다.
  • 같은 예산의 동시 차감, 유료·무료 원장 혼합 사용, 회원 식별자 중복, lock key가 같은 요청과 다른 요청을 나눠 검증했습니다.

재시작 기준 소멸은 페이지, 만료 안내는 발송 상태, 아카이브는 테이블 순서로 처리 위치를 확인합니다.

좌표나 필터, 로그인 상태가 달라지면 캐시에 값이 있어도 현재 요청에는 오래된 결과가 됩니다. 반대로 정상 스크롤까지 매번 다시 계산하면 검색과 랭킹 비용이 커집니다.

  • OpenSearch 후보 소스 9개를 병렬 조회하고, 소스마다 최대 200건을 받아 1차 랭킹 500건, 2차 랭킹 100건, 기본 페이지 20건으로 좁혔습니다.
  • Snapshot, 1차 랭킹 풀, 2차 랭킹 풀, 본 공고 저장소를 나눠 정상 스크롤은 가장 싼 캐시부터 사용하게 했습니다.
  • 세션 캐시는 30분 동안 재사용하되 좌표·필터 요청 지문이나 로그인 주체가 달라지면 관련 후보 풀을 무효화했습니다.
  • Redis 장애에서는 메모리 계산으로 우회하고, contentHash 중복 제거와 위치 기반 10km 반경 정책을 예외 경로에 반영했습니다.

판단 기준 스크롤, 후보 소진, 조건 변경, 로그인 전환, Redis 장애를 같은 cache miss로 취급하지 않았습니다.

fixture와 mock 초기화, WebMvc 요청, JPA 데이터 준비가 테스트마다 달라지면 무엇을 검증하는지보다 setup을 읽는 데 시간이 더 듭니다. 반복되는 준비 절차를 작은 도우미로 모으고 공통 동작 자체도 테스트했습니다.

  • fixture·mock·WebMvc·JPA·transaction 준비와 공통 진입점을 29개 구현 파일로 정리했습니다.
  • 기본 테스트, mock 테스트, WebMvc 테스트, JPA 테스트의 진입점을 나눠 필요한 컨텍스트만 선택하게 했습니다.
  • MockMvc 요청 빌더와 응답 검증, query string null 처리, JPA 초기화와 transaction 도우미를 제공했습니다.
  • 공통 모듈의 동작은 43개 테스트 소스로 확인하고 느린 테스트 제외, 기본 profile, JavaDoc까지 함께 정리했습니다.

팀에서 쓰는 방식 테스트 종류에 맞는 기반 클래스를 고르면 fixture와 mock, Spring 컨텍스트 준비 규칙을 같은 방식으로 사용할 수 있습니다.
