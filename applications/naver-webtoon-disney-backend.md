## 1. 지원 동기와 직무 적합성

이 공고에서 가장 눈에 들어온 부분은 두 가지입니다. 신규 글로벌 서비스를 준비 단계부터 출시까지 만든다는 점, 그리고 AI Harness를 개발 과정에 실제로 넣는다는 점입니다.

저는 약 3년간 Kotlin·Java와 Spring Boot로 이벤트·배치·검색·캐시 시스템을 설계하고 운영했습니다. 개인화 피드에서는 9개 검색 소스와 FPR·SPR·Re-Rank, Snapshot·FPR·SPR 캐시의 책임을 단계별 계약으로 분리했습니다. 팀 개발 절차는 15개 에이전트와 32개 스킬로 구조화하고, 설치 회귀 52개와 구조·행동 계약 17개로 검증했습니다.

Disney 디지털 코믹스 플랫폼에서도 같은 방식으로 기여하고 싶습니다. 출시 전에는 변경 범위가 분명한 구조와 검증 기준을 세우고, 출시 후에는 실패 지점과 운영 신호를 근거로 서비스를 개선하겠습니다.

## 2. 가장 관련 있는 프로젝트

가장 관련 있는 경험은 개인화 채용 피드의 검색·랭킹·캐시 구조를 설계한 일입니다. 한 오케스트레이터가 검색 쿼리와 점수 계산, Redis 처리까지 모두 알게 되면 작은 정책 변경도 전체 서빙 경로에 영향을 줍니다.

후보 조회를 9개 소스 포트로 분리하고 FPR·SPR·Re-Rank는 공통 pull 계약으로 연결했습니다. Snapshot·FPR·SPR 캐시는 상태 포트로 나눴고, query-api는 OpenSearch 연동만 담당하도록 제한했습니다. 의존 방향과 순환 금지는 ArchUnit으로 검증했습니다.

그 결과 한 검색 소스가 늦거나 실패해도 나머지 후보 조회는 계속할 수 있었고, Redis 장애 때도 별도 랭킹을 만들지 않고 캐시 없는 같은 흐름을 실행했습니다. 콘텐츠 탐색 정책이 자주 바뀌는 환경에서도 변경과 장애의 범위를 작게 유지할 수 있는 구조입니다.

## 3. AI Harness를 실제 업무에 적용한 경험

반복 개발에 AI를 적용하면서 코드 생성 속도보다 팀의 판단 기준을 남기는 일이 더 중요하다고 봤습니다. 그래서 코드 탐색·설계 검토·테스트·리뷰 절차를 재사용 가능한 자산으로 만들었습니다.

역할은 15개 에이전트, 실행 절차는 32개 스킬로 나눴습니다. manifest가 필요한 의존 자산까지 함께 설치하도록 구성하고, 설치 회귀 52개와 Codex 탐색 32개를 통과시켰습니다. 설치 여부만 확인하지 않도록 구조 계약 7개와 행동 계약 10개도 두었습니다.

대신 메모리와 자가 수정은 자동화하지 않았습니다. 평가 결과와 사람의 승인이 있어야 다음 버전에 반영됩니다. AI의 실행 속도는 활용하되 변경 권한과 검증 책임은 사람이 갖도록 만든 것이 제가 구축한 AI Harness의 핵심입니다.

## 4. 장애와 실패 처리를 주도한 경험

포인트 플랫폼에서는 발행 실패와 소비 중 실패가 하나의 상태로 남아, 장애가 나면 운영자가 로그와 원장을 처음부터 대조해야 했습니다.

이벤트 9종의 처리 단계를 6개 상태로 기록하고, 저장된 payload를 해당 업무 서비스에 다시 전달하는 복구 전략 8개를 구현했습니다. IN_PROGRESS 상태가 15분을 넘으면 이벤트 유형별로 집계해 Slack으로 알렸습니다.

전송 실패, 처리 실패, 장기 미처리를 나눈 뒤에는 전체 흐름을 처음부터 돌리지 않고 멈춘 위치의 업무만 다시 실행할 수 있었습니다. 이 경험 이후 장애 대응을 단순한 재시도 횟수의 문제가 아니라 실패 지점, 재시작 기준, 운영 신호를 함께 설계하는 문제로 보고 있습니다.

## 5. 가장 어려운 기술 문제를 해결한 경험

토스랩에서는 테스트 커버리지가 70%였지만 배포 안정성은 그만큼 높지 않았습니다. Mock 테스트가 JPA 변경 감지 오류를 놓쳤고, 그 영향으로 약 200만 건의 데이터를 바로잡아야 했습니다.

실제 영속성 컨텍스트에서 오류를 재현하는 구현 테스트를 추가해 원인을 확정한 뒤 보정 스크립트로 데이터를 수정했습니다. 이후 Drive-Service 테스트를 0개에서 160개 이상으로 늘리고 Spring Boot를 2.3.8에서 3.3.3으로 전환했습니다.

이 경험으로 커버리지 수치보다 운영과 같은 조건에서 실패를 재현하는 테스트가 더 중요하다는 기준을 갖게 됐습니다.

## 6. 협업하며 개발 기준을 만든 경험

웍스피어에서는 여러 모듈에서 Fixture·Mock·WebMvc·JPA 테스트 준비 코드가 반복됐고, 어떤 테스트를 CI에서 실행할지도 기준이 달랐습니다.

공통 동작을 test-support로 모은 뒤 테스트 목적에 따라 Base·Mock·WebMvc·JPA의 4개 진입점과 5개 공통 도구·정책으로 나눴습니다. 구축 당시 29개 구현 파일과 43개 테스트 소스에서 동작을 확인했고, 느린 동시성 테스트는 skip.slow.test 설정으로 분리했습니다. CI에서 테스트가 메모리 부족으로 멈추자 자원을 늘리는 대신 느린 테스트를 분리하고 Spring 컨텍스트의 재사용·초기화 방식을 바꿨습니다.

공통 코드만 만들고 끝내지 않았습니다. 테스트 작성 원칙과 CI 실행 범위를 함께 문서화해 팀이 같은 기준으로 테스트를 고르고 실패를 해석하도록 했습니다. 협업에서 중요한 것은 개인의 선호를 하나로 맞추는 일이 아니라 함께 판단할 기준을 만드는 일이라고 생각합니다.

## 7. 글로벌·컴플라이언스 경험을 묻는 경우

국가별 법령이나 글로벌 컴플라이언스를 직접 담당한 경험은 아직 없습니다. 대신 정책 변경의 영향을 제한하고 처리 근거를 남기는 설계를 꾸준히 다뤘습니다.

검색 소스별 필터·정렬·타임아웃은 독립 포트로 분리했고, 이벤트는 상태와 payload를 저장해 실패 단계를 추적할 수 있게 했습니다. 국가별 정책도 공통 로직 곳곳에 조건문으로 넣기보다 정책 단위로 격리하고, 변경 전후 결과를 계약 테스트와 운영 지표로 확인하겠습니다.

법무·기획과 적용 기준과 예외를 먼저 맞추고, 기존에 다뤄온 정책 격리·상태 추적·계약 검증 방식을 국가별 규정 반영에도 적용하겠습니다.
