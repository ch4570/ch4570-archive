# 백엔드 경력직 지원서 재료

- 대상: Java·Kotlin·Spring 기반 2~4년차 백엔드 경력직
- 근거: 현재 이력서·경력기술서·포트폴리오에서 확인되는 사실만 사용
- 사용법: 공고와 가까운 사례를 고른 뒤 회사와 문항에 맞게 다시 쓴다. 아래 문장을 여러 회사에 그대로 제출하지 않는다.

> 주력 사례는 포인트 이벤트 복구와 JPA 데이터 보정이다. 검색 경험은 공고가 직접 요구할 때만 꺼낸다. `[회사명]` 같은 표시가 남은 답변은 제출하지 않는다.

## 답변 초안

### 1. 한 줄 소개

```text
Java·Kotlin·Spring 기반 백엔드를 약 3년간 개발했고, 현재는 포인트 이벤트 복구와 원장·배치를 맡고 있습니다.
```

### 2. 짧은 자기소개

```text
현재 웍스피어(유)에서 JOBKOREA·Albamon 포인트 플랫폼의 이벤트 복구 기능과 원장·배치를 개발하고 있습니다. 이벤트 처리 상태를 나누고, 저장해 둔 payload로 실패한 업무만 다시 실행하도록 했습니다. 토스랩에서는 JPA 변경 감지 오류를 재현해 잘못 저장된 데이터를 보정했고, Spring Boot 3 전환과 함께 테스트를 보강했습니다.
```

### 3. 지원 동기를 쓰기 전 확인할 네 가지

- 이 회사의 어떤 서비스나 기능을 실제로 관심 있게 봤는가?
- 그 기능이 사용자나 사업에 어떤 영향을 주는가?
- 공고에 적힌 문제 중 내 경험과 가장 가까운 것은 무엇인가?
- 입사 후 처음 맡고 싶은 구체적인 문제는 무엇인가?

위 네 가지에 답하기 전에는 지원 동기를 완성하지 않는다. 회사 소개를 바꿔 끼운 문장보다, 관심을 가진 이유 하나와 관련 경험 하나만 쓰는 편이 낫다.

#### 경험 연결용 사실

- 이벤트·금액성 데이터 직무: `포인트 이벤트의 실패 상태와 재처리 기준을 나눴고, 동시 요청이 끝난 뒤 잔액과 원장의 정합성을 확인했습니다.`
- 검색·추천 직무: `개인화 피드의 후보 조회·랭킹·캐시 단계를 분리하고, 검색 소스별 실패 처리와 Redis 캐시를 우회하는 경로를 구현했습니다.`
- 테스트·플랫폼 직무: `웍스피어(유)에서 반복되던 테스트 준비 코드를 공통 모듈로 모으고, CI에서 실행할 테스트 범위를 문서화했습니다.`

### 4. Java·Kotlin과 Spring Boot 기반 백엔드 경험

```text
Java·Kotlin과 Spring Boot를 주력으로 약 3년간 백엔드를 개발해 왔습니다. 현재 웍스피어(유)에서는 Kafka 기반 내부 이벤트의 복구 기능과 포인트 원장·배치를 개발하고 있으며, 개인화 피드 개발에는 Redis와 OpenSearch를 사용하고 있습니다. 토스랩에서는 JPA 오류를 재현해 데이터를 보정했고, Drive-Service의 Spring Boot 2.3.8→3.3.3 전환 과정에서 테스트를 0개에서 160개 이상으로 보강했습니다.
```

### 5. 대표 프로젝트 — 이벤트 복구와 포인트 검증

```text
웍스피어(유)에서 포인트 이벤트의 발행 실패와 처리 실패가 같은 상태로 남아, 복구할 업무를 바로 구분하기 어려웠습니다. 처리 상태를 6개로 나누고, 저장해 둔 payload로 실패한 업무만 다시 실행하도록 바꿨습니다. 15분 이상 멈춘 이벤트는 Slack으로 알렸습니다. 별도로 포인트 거래 API에 동시 요청을 보낸 뒤 잔액과 원장을 함께 확인했습니다.
```

### 6. 대표 프로젝트 — 검색·랭킹·캐시 구조

```text
웍스피어(유)에서 개인화 피드를 개발하며 후보 조회, 랭킹, 캐시를 단계별로 분리했습니다. 검색 소스마다 타임아웃과 실패 처리 방식을 따로 정하고, Redis 캐시를 우회하는 경로도 구현했습니다. 모듈 간 의존 규칙은 ArchUnit 테스트로 검증했습니다.
```

### 7. 가장 어려운 기술 문제를 해결한 경험

```text
토스랩에서 Mock 테스트에서는 재현되지 않던 JPA 변경 감지 오류를 실제 영속성 컨텍스트에서 재현했습니다. 보정 스크립트를 작성해 잘못 저장된 데이터 약 200만 건을 바로잡았습니다.

```

### 8. 테스트 기반을 정리한 경험

```text
웍스피어(유)에서 Fixture 생성, Mock 설정, WebMvc·JPA 테스트에 공통으로 쓰던 준비 코드를 test-support 모듈로 모았습니다. Base·Mock·WebMvc·JPA별로 테스트 진입점을 나눠 각 테스트에 필요한 설정만 사용하도록 했고, 느린 동시성 테스트는 별도 옵션으로 분리했습니다. CI에서 기본으로 실행할 테스트 범위도 문서화했습니다.
```

> 협업 문항에는 그대로 쓰지 않는다. 사용한 팀·합의 과정·피드백처럼 다른 사람이 등장하는 사실을 확인한 뒤 보완한다.

### 9. 오케스트로 API·인증 경험 — 질문에 맞는 한 줄만 선택

- 인증·인가: `Spring Cloud Gateway, Spring Security, JWT를 사용해 인증·인가를 구현했습니다.`
- 개발 도구 연동: `SonarQube·Jenkins 연동 API를 개발했습니다.`
- 메트릭: `VM 메트릭 조회 API를 개발했습니다.`
- 검색: `조회 조건에 맞춰 OpenSearch 쿼리를 동적으로 생성하는 API를 개발했습니다.`

### 10. 일하는 방식 또는 강점

```text
재시도 로직부터 추가하기보다 실패한 단계와 복구할 데이터 범위를 먼저 확인합니다. 실제로 내부 이벤트는 실패 상태를 나눴고, 배치는 다시 시작할 위치를 기록했습니다. 동시 요청 테스트에서는 잔액과 원장의 정합성을 함께 확인했습니다.
```

## 제출 전 체크리스트

- 공고에서 반복되는 기술·업무 키워드 세 개를 고르고, 가장 가까운 사례 하나를 정한다.
- 지원 동기를 쓰기 전 위의 네 질문에 구체적으로 답했는지 확인한다.
- 한 문항에는 대표 프로젝트 하나만 사용하고 같은 수치를 여러 답변에서 반복하지 않는다.
- 공고가 강조하는 키워드는 이력서에 실제 근거가 있을 때만 같은 표현으로 맞춘다.
- `RESTful 설계`, `대용량 트래픽`, `결제 연동`, `Kubernetes 운영`, `non-blocking`, `글로벌 컴플라이언스`를 직접 수행한 것처럼 표현하지 않는다.
- 글자 수 제한에 맞춰 배경보다 행동·검증 결과를 먼저 남긴다.
- 제출 전 회사명, 서비스명, 지원 직무명, 링크 공개 상태와 Vercel 배포본의 문구·PDF 페이지 수를 다시 확인한다.
- 이메일로 직접 제출할 때는 지원 공고가 요구하는 전화번호를 최종 PDF에 포함했는지 확인한다.

## 지원 전 확보할 실제 수치

아래 값은 사내 자료나 본인 기록으로 확인된 경우에만 이력서와 답변에 추가한다.

- 이벤트: 일 처리량, 실패 건수, 재처리 건수, 복구 전후 소요 시간, 알림 이후 대응 시간
- 배치: 대상 데이터량, 실행 시간, chunk 크기, 실패 후 재시작 시간
- API·캐시: RPS, p95/p99, 캐시 hit rate, Redis 장애 시 fallback 검증 결과
- 테스트·CI: 전체 실행 시간, 메모리 사용량 전후, 실패율, 실제 사용 모듈·팀 수
- 역할: 팀 규모, 본인 담당 범위, 직접 결정한 설계, 배포·운영 여부

확인할 수 없는 값은 추정해 넣지 않는다. 대신 `무엇을 구현했고 어떤 상태를 검증했는지`까지로 범위를 한정한다.

## 이직 사유 준비 메모

지원서에 그대로 붙이는 문장이 아니라 면접 전 사실관계를 정리하는 용도다.

- `웍스피어(유) 소속`과 `JOBKOREA·Albamon 플랫폼 업무`를 구분해 설명한다.
- 각 회사에서 이직한 이유를 회사 탓이 아닌 역할·성장·업무 연속성 관점의 사실로 정리한다.
- 짧은 재직 기간이 반복된 이유와 이번에는 어떤 문제를 장기적으로 맡고 싶은지 연결한다.
- 현재 이직을 고려하는 이유와 최소 2년 이상 축적하고 싶은 전문 영역을 일관되게 설명한다.
- 실제 사유를 정리하기 전에는 계약 종료·프로젝트 종료·조직 변경을 추정해 쓰지 않는다.
