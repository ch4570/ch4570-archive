# 백엔드 경력직 지원서 재료

- 대상: Java·Kotlin·Spring 기반 2~4년차 백엔드 경력직
- 근거: 현재 이력서·경력기술서·포트폴리오에서 확인되는 사실만 사용
- 사용법: 공고와 가까운 사례를 고른 뒤 회사와 문항에 맞게 다시 쓴다. 아래 문장을 여러 회사에 그대로 제출하지 않는다.

> 주력 사례는 포인트 이벤트 복구와 JPA 데이터 보정이다. 검색 경험은 공고가 직접 요구할 때만 꺼낸다. `[회사명]` 같은 표시가 남은 답변은 제출하지 않는다.

## 답변 초안

### 1. 한 줄 소개

```text
Java·Kotlin·Spring으로 백엔드를 개발해 온 약 3년 경력의 엔지니어로, 최근에는 포인트 이벤트 복구와 배치·원장을 맡았습니다.
```

### 2. 짧은 자기소개

```text
현재 웍스피어(유) 소속으로 JOBKOREA·Albamon 포인트 플랫폼의 이벤트 실패 복구와 원장·배치를 개발하고 있습니다. 이벤트 처리 상태를 나누고 저장된 payload로 실패 단계에 맞는 업무만 다시 실행하도록 했습니다. 이전 직장에서는 JPA 변경 감지 오류로 잘못 저장된 데이터를 보정하고 테스트 기반을 다시 만들었습니다.
```

### 3. 지원 동기는 먼저 네 가지를 확인한다

- 이 회사의 어떤 서비스나 기능을 실제로 관심 있게 봤는가?
- 그 기능이 사용자나 사업에 어떤 영향을 주는가?
- 공고에 적힌 문제 중 내 경험과 가장 가까운 것은 무엇인가?
- 입사 후 처음 맡고 싶은 구체적인 문제는 무엇인가?

네 답을 확인하기 전에는 지원 동기를 완성하지 않는다. 회사 소개를 바꿔 끼운 문장보다, 관심을 가진 이유 하나와 관련 경험 하나만 쓰는 편이 낫다.

#### 경험 연결용 사실

- 이벤트·금액성 데이터 직무: `포인트 이벤트의 실패 상태와 재처리 기준을 나누고, 동시 요청 뒤 원장이 맞는지 확인했습니다.`
- 검색·추천 직무: `검색 소스·랭킹·캐시의 책임을 나누고, 소스별 실패 처리와 Redis 캐시 우회 경로를 구현했습니다.`
- 테스트·플랫폼 직무: `웍스피어(유)에서 모듈마다 반복되던 테스트 준비 코드를 공통 모듈로 옮기고 CI 실행 기준을 정리했습니다.`

### 4. Java·Kotlin과 Spring Boot 기반 백엔드 경험

```text
약 3년 동안 Java·Kotlin과 Spring Boot로 백엔드를 개발했습니다. 현재는 Kafka로 처리하는 내부 이벤트의 상태와 복구 경로, Spring Batch, 포인트 원장을 맡고 있으며 Redis와 OpenSearch도 사용합니다. 이전 직장에서는 JPA 오류를 재현해 데이터를 보정하고 Spring Boot 3 전환을 진행했습니다.
```

### 5. 대표 프로젝트 — 이벤트 신뢰성과 포인트 정합성

```text
포인트 이벤트의 발행 실패와 소비 실패가 같은 상태로 남아, 어떤 업무부터 복구할지 구분하기 어려웠습니다. 처리 단계를 6개 상태로 나누고 저장된 payload로 실패한 업무만 다시 실행하도록 바꿨습니다. 15분 이상 멈춘 이벤트는 Slack으로 알렸고, 포인트 거래는 동시성 테스트로 잔액과 원장이 맞는지 확인했습니다.
```

### 6. 대표 프로젝트 — 검색·랭킹·캐시 구조

```text
개인화 피드의 후보 조회·랭킹·캐시 책임을 나눴습니다. 검색 소스별 타임아웃·실패 처리와 Redis 캐시 우회 경로를 구현했고, 모듈 간 의존 규칙은 ArchUnit 테스트로 확인했습니다.
```

### 7. 가장 어려운 기술 문제를 해결한 경험

```text
Mock 중심 테스트가 JPA 변경 감지 오류를 놓쳐 약 200만 건의 데이터가 잘못 저장됐습니다.

실제 영속성 컨텍스트에서 오류를 재현해 원인을 확인하고 보정 스크립트로 수정했습니다.

```

### 8. 테스트 기반을 정리한 경험

```text
웍스피어(유)에서 모듈마다 반복되던 Fixture·Mock·WebMvc·JPA 테스트 준비 코드를 test-support로 모았습니다. 테스트 목적에 따라 시작점을 나누고, 느린 동시성 테스트는 별도 옵션으로 분리했습니다. 어떤 테스트를 CI에서 실행할지도 문서로 정리했습니다.
```

> 협업 문항에는 그대로 쓰지 않는다. 사용한 팀·합의 과정·피드백처럼 다른 사람이 등장하는 사실을 확인한 뒤 보완한다.

### 9. API·인증 경험 — 질문에 맞는 한 줄만 선택

- 인증·인가: `Spring Cloud Gateway와 Spring Security·JWT로 인증·인가를 구현했습니다.`
- 개발 도구 연동: `SonarQube·Jenkins 연동 API를 개발했습니다.`
- 메트릭: `VM 메트릭 조회 API를 개발했습니다.`
- 검색: `조건에 따라 OpenSearch 쿼리를 만드는 조회 API를 개발했습니다.`

### 10. 일하는 방식 또는 강점

```text
장애가 나면 재시도부터 붙이기보다 어디서 멈췄고 어떤 데이터부터 다시 처리해야 하는지 먼저 확인합니다. 이 원칙으로 이벤트 실패 상태와 배치 재시작 지점을 나누고, 동시 요청 뒤에는 잔액과 원장이 맞는지 확인했습니다.
```


## 제출 전 체크리스트

- 공고에서 반복되는 기술·업무 키워드 세 개를 고르고, 가장 가까운 사례 하나를 정한다.
- 지원 동기를 쓰기 전 위의 네 질문에 구체적으로 답했는지 확인한다.
- 한 문항에는 대표 프로젝트 하나만 사용하고 같은 수치를 여러 답변에서 반복하지 않는다.
- 공고가 강조하는 키워드는 이력서에 실제 근거가 있을 때만 같은 표현으로 맞춘다.
- `RESTful 설계`, `대용량 트래픽`, `결제 연동`, `Kubernetes 운영`, `non-blocking`, `글로벌 컴플라이언스`를 직접 수행한 것처럼 표현하지 않는다.
- 글자 수 제한에 맞춰 배경보다 행동·검증 결과를 먼저 남긴다.
- 제출 전 회사명, 서비스명, 지원 직무명, 링크 공개 상태와 Vercel 배포본의 문구·PDF 페이지 수를 다시 확인한다.

## 지원 전 확보할 실제 수치

아래 값은 사내 자료나 본인 기록으로 확인된 경우에만 이력서와 답변에 추가한다.

- 이벤트: 일 처리량, 실패 건수, 재처리 건수, 복구 전후 소요 시간, 알림 이후 대응 시간
- 배치: 대상 데이터량, 실행 시간, chunk 크기, 실패 후 재시작 시간
- API·캐시: RPS, p95/p99, 캐시 hit rate, Redis 장애 시 fallback 검증 결과
- 테스트·CI: 전체 실행 시간, 메모리 사용량 전후, 실패율, 실제 사용 모듈·팀 수
- 역할: 팀 규모, 본인 담당 범위, 직접 결정한 설계, 배포·운영 여부

확인할 수 없는 값은 추정해 넣지 않는다. 대신 `무엇을 구현했고 어떤 상태를 검증했는지`까지로 범위를 한정한다.

## 이직 사유 준비 메모

지원서에 그대로 붙이는 문장이 아니라 면접 전 사실관계를 정리하는 용도다.

- `웍스피어(유) 소속`과 `JOBKOREA·Albamon 플랫폼 업무`를 구분해 설명한다.
- 각 회사에서 이직한 이유를 회사 탓이 아닌 역할·성장·업무 연속성 관점의 사실로 정리한다.
- 짧은 재직 기간이 반복된 이유와 이번에는 어떤 문제를 장기적으로 맡고 싶은지 연결한다.
- 현재 이직을 고려하는 이유와 최소 2년 이상 축적하고 싶은 전문 영역을 일관되게 설명한다.
- 실제 사유를 정리하기 전에는 계약 종료·프로젝트 종료·조직 변경을 추정해 쓰지 않는다.
